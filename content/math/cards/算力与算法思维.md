# 算力与算法思维

> **Intuitive 卡片不教你"怎么算"，而是教你"算什么"。** 高考不会让你写代码——但"算法思维"会让你在数学题中识别出"这题可以用程序化的思路解决"，从而把枯燥的手算变成清晰的流程。

---

## 视角层（像什么）

### 核心隐喻：大厨的菜谱——不是做一道菜，而是掌握一套做菜的流程

**算力与算法思维 = 大厨手中的菜谱——菜谱不是某一道菜，而是"如何一步步做出菜"的通用流程**

```
  菜谱（算法）                    菜（结果）
┌────────────────┐           ┌────────────────┐
│ 1. 准备食材    │           │                │
│ 2. 切菜        │   执行    │   番茄炒蛋     │
│ 3. 热油        │  ─────→  │                │
│ 4. 炒蛋        │           │  （无论谁按    │
│ 5. 加番茄      │           │   菜谱做，都   │
│ 6. 调味        │           │   能做出这道菜）│
└────────────────┘           └────────────────┘

  菜谱 = 算法的核心特征：
  ✅ 步骤明确（每一步说什么就做什么）
  ✅ 顺序固定（不能先炒再放油）
  ✅ 可重复（不同人执行同样的菜谱，结果一致）
  ✅ 有限步（不会永远做不完）
```

**菜谱思维的三条铁律：**

| 铁律 | 做菜场景 | 违反的后果 |
|------|---------|-----------|
| **步骤分解**：大问题切成小步骤 | 不会"做菜"但会"切菜""热油""翻炒" | 面对复杂问题无从下手 |
| **确定性**：每一步的结果是确定的 | "热油到七成"是明确的判据 | 算法有歧义，不同人执行结果不同 |
| **停机条件**：知道什么时候停 | 炒到蛋凝固就出锅，不是永远炒下去 | 无限循环，算不停 |

> **一句话直觉：** 算力是"能算多快"，算法思维是"知道按什么顺序算"——高考不要求会写代码，但要求你能把一个问题拆成清晰的步骤序列，然后按顺序执行。

### 动态想象：二分查找——算法思维的经典案例

```
  猜数字游戏：我心里想了一个 1-100 之间的数，你猜是什么。
  每次猜完我告诉你是"大了"还是"小了"。

  普通人：1, 2, 3, 4...（最坏猜 100 次）
  算法思维：50 → 大了？→ 25 → 小了？→ 37 → ...

  二分法：
  每次猜中间值，范围每次都缩小一半
  100 → 50 → 25 → 12 → 6 → 3 → 1
  最坏只需要 7 次！

  这就是算法思维的威力——
  不是"算得更快"，而是"用更聪明的方式算"
```

### 反直觉警示

| 翻车点 | 直觉告诉你 | 实际真相 |
|--------|-----------|---------|
| **"我手算就行"** | "这题不就是…吗，不用算法" | 当数据量从 10 变到 10000 时，手算不可能——需要算法思维设计出可扩展的方法 |
| **"算法 = 代码"** | "我不会编程，所以我不用算法思维" | 算法思维是"流程化思考"，不一定要写成代码。二分法、递推、筛法都是算法思维 |
| **"结果对就行"** | "反正答案对了，方法无所谓" | 同样的答案，O(n) 算法和 O(2^n) 算法在数据大时差距是天壤之别 |

## 联结层（连什么）

### 跨学科锚点

**1. 计算机科学：时间复杂度的直觉**
```
在通讯录里找一个人：
  方法A（一页页翻）：O(n) —— 100 页 ≈ 100 次
  方法B（二分法翻）：O(log n) —— 100 页 ≈ 7 次
  方法C（直接翻到姓氏首字母）：O(1) —— 不管多少页，一次到位
```

**2. 生活：整理房间的算法**
```
收拾乱房间：
  算法1：看到什么收什么（随机）→ 永远收不完
  算法2：分类→逐类归位→扫地拖地→检查（有序）→ 2 小时搞定
好的算法 = 好的流程设计
```

### 向 Formal 的桥梁

| 数学场景 | 算法思维 | Formal 着陆点 |
|---------|---------|--------------|
| 解方程 f(x)=0 | 二分法不断缩小区间 | 二分法.md, 连续方程 |
| 求 √2 的近似值 | 牛顿迭代 x_{n+1}=(x_n+2/x_n)/2 | 牛顿迭代.md |

### 概念网络坐标

```
  复杂数学问题
      │
      ▼
  问：能否分解为重复的简单步骤？
      │
      ├── 是 → 设计算法流程 → 递推/二分/迭代
      └── 否 → 寻找精巧的数学技巧
```

## 迁移层（何时调用）

### 三类场景判定表

| 场景 | 识别信号 | 拆包收益 |
|:---|:---|:---|
| 🎭 **包装** | "设计一种方法求…""按以下规则操作…" | 识别出这是在描述算法流程，画流程图 |
| 🔀 **嵌套** | 多层判断+循环+条件分支 | 拆成判断树和循环体，逐层分析 |
| 🆕 **新定义** | "定义运算：从 a₁ 出发，反复执行…" | 这就是在定义迭代算法，试试用具体数字跑一遍 |

### 拆包方向标（10 秒流程）

```
Step 1：识别是否需要算法思维（3 秒）
  • 问题涉及"重复操作""逐步逼近""大量数据"？
  • 手算太繁琐？→ 需要算法思维

Step 2：设计算法步骤（4 秒）
  • 输入什么？每步做什么？输出什么？
  • 什么时候停止？

Step 3：锁定 Formal 着陆点（3 秒）
  • 确定用递推/二分/迭代/筛法
```

### 直觉陷阱清单

| 陷阱 | 何时失效 | 纠错方式 |
|------|---------|---------|
| **算法死循环** | 没有设计停机条件 | 每一步后检查：是否已满足停止条件？ |
| **效率盲区** | 能算出来但方法太慢 | 问自己：这个方法在最坏情况下需要几步？能否更快？ |
| **精度陷阱** | 迭代逼近时不知道什么时候精度够了 | 设定明确的精度阈值（如 |x_{n+1}-x_n|<10^{-6}） |

## 一句话总结

> **算法思维不是会写代码，而是会"流程化思考"——面对复杂问题，把它拆成确定性的、可重复的、有限步的步骤序列，然后按顺序执行。高考里的二分法、牛顿迭代、递推——本质上都是算法思维。**
